# Certificate Management for Codex <-> F5 AI Guardrails

## Overview

When running the Codex <-> F5 AI Guardrails smoke test from Python, some Windows users may see an SSL certificate validation error even though the same URL works in their browser.

A typical error looks like this:

```text
SSLCertVerificationError:
certificate verify failed: unable to get local issuer certificate
```

This usually means Python reached the Guardrails API endpoint, received a certificate, but could not build a trusted certificate chain from that certificate back to a trusted root CA.

This is common in corporate environments that use TLS inspection products such as Zscaler, Netskope, Blue Coat, Fortinet, Palo Alto, or similar perimeter security tools.

---

## Why this happens

Browsers on managed Windows devices often trust corporate root certificates because those certificates are installed into the Windows certificate store by IT policy.

Python is different.

The Python `requests` library typically validates HTTPS certificates using the `certifi` CA bundle, not always the Windows certificate store. This integration installs and uses the Python `truststore` package when available so Windows can use the native Windows Cert Store. If `truststore` is not installed or is disabled, a connection can work in Chrome or Edge while the same connection fails in Python.

In a TLS inspection environment, the flow often looks like this:

```text
Python client -> Zscaler / TLS inspection proxy -> Guardrails API
```

The browser may show a certificate for the correct hostname, such as:

```text
us1.calypsoai.app
```

but the issuer may be a corporate security CA, such as:

```text
Zscaler
```

or another internal perimeter security CA.

That corporate CA must be trusted by Python before `requests` will accept the connection.

---

## Do not disable certificate verification

You may see examples that use:

```python
verify=False
```

Do not use that as a permanent fix.

Disabling certificate verification means the client is no longer validating that it is talking to the intended server. It hides the problem instead of fixing the trust path.

The correct fix is one of these:

1. On Windows, use the native Windows Cert Store through `truststore`.
2. If system-store trust is unavailable or disabled, provide Python with a CA bundle that contains both the normal public CA roots used by `certifi` and the corporate TLS inspection root CA.

## Step 1: Confirm what Python is using for certificates

Run these commands from PowerShell:

```powershell
python -c "import certifi; print(certifi.where())"
```

```powershell
python -c "import requests; print(requests.certs.where())"
```

```powershell
python -c "import ssl; print(ssl.get_default_verify_paths())"
```

You will likely see a path similar to:

```text
C:\Users\user\AppData\Local\Programs\Python\Python314\Lib\site-packages\certifi\cacert.pem
```

That file is the default CA bundle used by Python `requests`.

Do not edit this file directly. It can be overwritten when `certifi` is upgraded.

---

## Step 2: Use the Windows Cert Store when available

The installer installs `truststore` with the hook dependencies. On Windows, `F5_GUARDRAILS_USE_SYSTEM_CERT_STORE=auto` uses the Windows Cert Store when `truststore` is available.

For the current PowerShell session, you can force this path:

```powershell
$env:F5_GUARDRAILS_USE_SYSTEM_CERT_STORE = "true"
python .\smoketest.py --tls-diagnostics
```

The TLS diagnostics should include lines like:

```text
System cert store:       enabled via truststore
F5 use system store:     true
```

To persist the setting for Codex GUI:

```powershell
[Environment]::SetEnvironmentVariable(
  "F5_GUARDRAILS_USE_SYSTEM_CERT_STORE",
  "true",
  "User"
)
```

Fully quit and relaunch Codex GUI after changing persistent environment variables. Existing GUI processes do not inherit newly changed user environment variables.

If this succeeds, you do not need to export a `.cer` file, convert it to PEM, or set `REQUESTS_CA_BUNDLE`.

If diagnostics show `truststore is not installed`, run:

```powershell
python -m pip install truststore
```

or rerun `install.ps1`.

## Step 3: Check whether the browser is seeing a Zscaler-issued certificate

In Chrome or Edge:

1. Open the Guardrails API site or application endpoint in the browser.
2. Open Developer Tools.
3. Go to the **Security** tab.
4. Click **View certificate**.
5. Review the certificate details.

If the certificate is issued by Zscaler or another corporate security CA, then the browser is going through TLS inspection.

The certificate may show:

```text
Issued To:
    Common Name: us1.calypsoai.app

Issued By:
    Organization: Zscaler Inc.
```

or something similar.

This means the browser trusts the corporate CA, but Python may not.

---

## Step 4: Locate the corporate CA in the Windows certificate store

Run this from PowerShell to search trusted root certificates:

```powershell
Get-ChildItem Cert:\CurrentUser\Root, Cert:\LocalMachine\Root |
  Where-Object { $_.Subject -match "Zscaler|Perimeter|Calypso|F5" -or $_.Issuer -match "Zscaler|Perimeter|Calypso|F5" } |
  Select-Object Subject, Issuer, Thumbprint, NotAfter
```

Also check intermediate certificates:

```powershell
Get-ChildItem Cert:\CurrentUser\CA, Cert:\LocalMachine\CA |
  Where-Object { $_.Subject -match "Zscaler|Perimeter|Calypso|F5" -or $_.Issuer -match "Zscaler|Perimeter|Calypso|F5" } |
  Select-Object Subject, Issuer, Thumbprint, NotAfter
```

Look for a trusted root or intermediate certificate related to Zscaler or your organization’s perimeter security service.

---

## Step 5: Fallback: Export the corporate root CA from the browser

Use this fallback only when the Windows Cert Store path is unavailable, disabled, or not accepted by your organization’s Python environment policy.

In Chrome or Edge:

1. Open the site.
2. Open Developer Tools.
3. Go to **Security**.
4. Click **View certificate**.
5. Go to **Certification Path**.
6. Select the parent or root certificate above the site certificate.
7. Click **View Certificate**.
8. Go to **Details**.
9. Click **Copy to File**.
10. Choose **Base-64 encoded X.509 (.CER)**.
11. Save the file as:

```text
%USERPROFILE%\.codex\certs\zscaler-root.cer
```

Create the folder first if needed:

```powershell
New-Item -ItemType Directory -Force "$env:USERPROFILE\.codex\certs"
```

If the exported file is Base-64 encoded, it is PEM-compatible. You can copy or rename it to `.pem`:

```powershell
Copy-Item `
  "$env:USERPROFILE\.codex\certs\zscaler-root.cer" `
  "$env:USERPROFILE\.codex\certs\zscaler-root.pem"
```

---

## Step 6: Fallback: Create a combined CA bundle

The safest approach is to combine the standard Python `certifi` bundle with the corporate root CA.

This preserves normal public CA trust while adding the corporate TLS inspection CA.

Run this in PowerShell:

```powershell
$CertifiBundle = python -c "import certifi; print(certifi.where())"
$CombinedBundle = "$env:USERPROFILE\.codex\certs\combined-ca-bundle.pem"

Copy-Item $CertifiBundle $CombinedBundle -Force

Add-Content $CombinedBundle ""
Get-Content "$env:USERPROFILE\.codex\certs\zscaler-root.pem" | Add-Content $CombinedBundle
```

This creates:

```text
%USERPROFILE%\.codex\certs\combined-ca-bundle.pem
```

---

## Step 7: Fallback: Point Python requests at the combined bundle

For the current PowerShell session:

```powershell
$env:REQUESTS_CA_BUNDLE = "$env:USERPROFILE\.codex\certs\combined-ca-bundle.pem"
$env:SSL_CERT_FILE = "$env:USERPROFILE\.codex\certs\combined-ca-bundle.pem"
```

Then test again:

```powershell
python .\smoketest.py --tls-diagnostics
```

If the smoke test succeeds, you can make these environment variables persistent for the current user.

```powershell
[Environment]::SetEnvironmentVariable(
  "REQUESTS_CA_BUNDLE",
  "$env:USERPROFILE\.codex\certs\combined-ca-bundle.pem",
  "User"
)

[Environment]::SetEnvironmentVariable(
  "SSL_CERT_FILE",
  "$env:USERPROFILE\.codex\certs\combined-ca-bundle.pem",
  "User"
)
```

Close and reopen PowerShell, then confirm:

```powershell
echo $env:REQUESTS_CA_BUNDLE
echo $env:SSL_CERT_FILE
python .\smoketest.py --tls-diagnostics
```

A successful TLS diagnostic should show something like:


## Optional: Use OpenSSL to inspect the certificate chain

If OpenSSL is available, run:

```powershell
openssl s_client `
  -connect www.us1.calypsoai.app:443 `
  -servername www.us1.calypsoai.app `
  -showcerts `
  -verify_return_error
```

If the machine is behind TLS inspection, the issuer may be Zscaler or another corporate CA.

If the chain fails verification, OpenSSL will usually show where the chain is incomplete or untrusted.

## Optional: Use SSLKEYLOGFILE for packet troubleshooting

For deep troubleshooting with Wireshark, you can set:

```powershell
$env:SSLKEYLOGFILE = "$env:TEMP\python-sslkeys.log"
python .\smoketest.py
```

The resulting key log file can be loaded into Wireshark to decrypt the TLS session, assuming the Python SSL stack supports key logging in that environment.

Treat the key log file as sensitive. It can decrypt captured traffic for that session.

## Common failure patterns

### Python fails, browser works

Likely cause:

```text
Browser trusts the corporate CA, but Python requests does not.
```

Fix:

```text
On Windows, use F5_GUARDRAILS_USE_SYSTEM_CERT_STORE=true. If that is unavailable, create a combined CA bundle and set REQUESTS_CA_BUNDLE / SSL_CERT_FILE.
```

### Error: unable to get local issuer certificate

Likely cause:

```text
The issuing CA is not in Python's trusted CA bundle.
```

Fix:

```text
On Windows, use the Windows Cert Store through truststore. If that is unavailable, add the corporate CA to a combined bundle.
```

### Error only happens on corporate network or VPN

Likely cause:

```text
TLS inspection is only active on managed networks, VPN, or corporate egress paths.
```

Fix:

```text
Use the Windows Cert Store through truststore or a corporate CA bundle when connected through that path.
```

### Smoke test says cleared but also logs an SSL error

Likely cause:

```text
The client may be failing open and returning a default cleared result even though the scan request failed.
```

Fix:

```text
Make the smoke test fail if result.is_error is True, even if outcome is cleared.
```

## Recommended repository behavior

For this project, the installer or quickstart should:

1. Install `truststore` with the Python hook dependencies.
2. Use `F5_GUARDRAILS_USE_SYSTEM_CERT_STORE=auto` by default so Windows can use the native Cert Store.
3. Detect whether `REQUESTS_CA_BUNDLE` or `SSL_CERT_FILE` is already set.
4. Print the active system-store status and Python `certifi` bundle path.
5. Run `smoketest.py --tls-diagnostics` when troubleshooting is needed.
6. Avoid modifying the `certifi` bundle in-place.
7. Avoid using `verify=False` except as a temporary diagnostic.
8. Document how to create a combined corporate CA bundle as a fallback.

---

## Recommended file locations

Use:

```text
%USERPROFILE%\.codex\certs\zscaler-root.pem
%USERPROFILE%\.codex\certs\combined-ca-bundle.pem
```

These locations keep certificate configuration close to the Codex integration while avoiding changes to the Python installation.

## Security note

The corporate root CA allows Python to trust certificates generated by the corporate TLS inspection platform. That is expected in managed enterprise environments, but it is also sensitive trust material.

Only use root CA certificates provided by your organization or exported from a trusted managed browser/device. Do not download random CA certificates from the internet and add them to your trusted bundle.
